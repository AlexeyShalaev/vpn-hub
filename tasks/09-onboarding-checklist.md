## 9. Онбординг-чеклист для нового owner

**Категория** UX · **Сложность** M · **Зависимости** №11 (i18n — строки через t()); переиспользуется в №10 (Home)

### Зачем
Пустой экран не ведёт нового владельца за руку: неочевидно, что путь к работающему VPN — «добавь сервер → установи VPN → создай группу → пригласи → выдай доступ». Чеклист с реальным прогрессом снижает порог первого успеха.

### Что уже есть в коде
- Точка входа owner — `frontend/src/screens/Servers.tsx`.
- Данные для шагов уже отдаются существующими query-хелперами `lib/queries.ts`: `listServers` (сервера с `protocols[].state`), `listGroups` (группы с `members[]` и `access.pools`/`access.servers`).
- i18n (`lib/i18n.ts`, `useT()`), навигация `useNav().go`, UI-кит `components/ui.tsx`.

### Дизайн решения
Самостоятельный компонент `frontend/src/components/OnboardingChecklist.tsx` (чтобы №10/Home переиспользовала его сверху). Прогресс ВЫЧИСЛЯЕТСЯ из данных, флаги не хранятся:
- `hasServer` — `servers.length > 0`;
- `hasInstalled` — есть протокол `state === "installed"` хоть на одном сервере;
- `hasGroup` — `groups.length > 0`;
- `hasMember` — в какой-то группе есть участник с `id !== me.id` (кто-то кроме владельца);
- `hasAccess` — у группы непустой `access.pools` или `access.servers`.

5 шагов (server/install/group/invite/access), каждый — со статусом done и переходом `go(...)` в соответствующий раздел (serverForm/servers/groups/access). Данные берутся через `useQuery(["servers"]|["groups"])` — те же ключи, что и остальное приложение (кэш переиспользуется, обновляется при инвалидации). Все строки — через `t("onboarding.*")`, ключи заведены в ru И en (паритет проверяет tsc).

Показ: встроен вверху `Servers.tsx` (owner-точка входа). Когда все шаги сделаны (`done === steps.length`) — компонент возвращает `null` (исчезает). Member не затрагивается (Servers — owner-экран).

### Реализовано в этом коммите
- `OnboardingChecklist.tsx` (169 строк): вычисление шагов + презентация (карточка с прогрессом «done/total», кликабельные шаги, галочки).
- 13 ключей `onboarding.*` в ru+en (`lib/i18n.ts`).
- Встраивание вверху `Servers.tsx`.

### Как тестировать
tsc (`make front-lint`) — валидность ключей t() и паритет ru/en; `npm run build` — сборка. Логика шагов — чистые булевы предикаты над данными query; проверяется визуально по мере заполнения (нет сервера → все шаги пусты; появился сервер → первый шаг done; всё готово → чеклист исчезает).

### Риски и подводные камни
- `useQuery` без данных (загрузка) — шаги трактуются как «не сделано»; чеклист мигнёт до подгрузки. Приемлемо.
- Совпадение ключей кэша с остальным приложением обязательно (`["servers"]`, `["groups"]`) — иначе двойной запрос.

### Критерии приёмки
- Владелец видит чеклист вверху главного экрана с корректным прогрессом; клик по шагу ведёт в нужный раздел; при полном прохождении чеклист исчезает; строки переключаются RU/EN.
- tsc и сборка зелёные.
