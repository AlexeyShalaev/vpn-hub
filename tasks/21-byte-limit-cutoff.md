## 21. Лимиты — Этап 3b: честная отсечка по трафику (suspend/resume)

**Категория** Лимиты · **Сложность** XL · **Зависимости** Этап 3a ([tasks/20](20-byte-limits.md))

### Зачем
Байт-лимит должен реально отсекать доступ, а не только блокировать выдачу новых конфигов. При
превышении лимита за период доступ пользователя на сервере приостанавливается, при возврате под лимит
(в т.ч. после сброса периода или повышения лимита) — возвращается. Требования пользователя: **не
заставлять перевставлять конфиг** (материал сохраняется) и **не отдавать освободившийся слот** новому
конфигу (иначе на resume конфликт).

### Механизмы suspend/resume (по протоколам — сохраняют материал и слот)
- **awg/awg_legacy**: `iptables DROP` клиентского `/32` в FORWARD (в обе стороны). Пир НЕ трогаем —
  IP-слот и порядок аллокации (`ipalloc` = последний AllowedIPs+1) сохранены. resume снимает правило.
- **xray/xray_xhttp**: убрать uuid из живого `server.json` (`clientsTable` не трогаем); resume — вернуть
  ТОТ ЖЕ uuid. uuid глобально уникален → новый конфиг «слот» не займёт.
- **hysteria2**: убрать строку `cid password` из файла токенов; resume — вернуть её.
- **outline**: per-key `data-limit` 0 байт (ключ/URL целы); resume — снять лимит.
- **openvpn**: обратимый `disable` через `client-config-dir` (ccd/<CN>); серт цел. Включение ccd
  идемпотентно (рестарт контейнера только при первом включении). `disable` вступает в силу на следующем
  коннекте клиента (активную долгую сессию не рвём, чтобы не трогать остальных).

### Реализация
- **provisioners.\***: `suspend_client(ssh, material)` / `resume_client(ssh, material)` (+ в base Protocol).
- **ProvisioningService**: `material_from_config` (из DeviceConfig, с расшифровкой секрета),
  `suspend_configs` / `resume_configs` — группировка по серверу, одна SSH-сессия, best-effort →
  множество реально применённых `client_id` (по нему обновляем статус).
- **SyncService**: фаза 5 `_reconcile_byte_limits(server)` — после сбора трафика: по каждому
  пользователю сервера с байт-лимитом сравнить usage (`period_usage`) vs `effective_byte_limit` за
  текущий период; `active`+over → suspend, `suspended`+under → resume; статус `DeviceConfig.status="suspended"`.
- **Пер-серверный лок sync** (`_SERVER_SYNC_LOCKS`): scheduled `run_tick` и ручной `POST /servers/{id}/sync`
  больше не идут параллельно на одном сервере → закрыт double-count дельт (отмечен в 3a) перед отсечкой.
- **UI**: на конфиге устройства статус «приостановлен (лимит трафика)» (Devices).

### Взаимодействие с 3a
`_find_config` матчит только `status="active"`, поэтому suspended-конфиг «не находится» → при попытке
перевыдачи срабатывает мягкий байт-блок 3a (пользователь над лимитом) → обойти отсечку перевыпуском нельзя.
После сброса периода usage за новый период = 0 → resume вернёт статус active → `_find_config` снова матчит.

### Тесты
- `test_suspend_resume.py` — команды каждого провизионера на fake SSH (iptables/uuid/строка/data-limit/ccd).
- `test_byte_limit_reconcile.py` — решение suspend/resume на реальном SQLite с фейковым ProvisioningService
  (over→suspend, under→resume, без лимита→no-op).

### Осталось / заметки
- Живой suspend/resume на реальных пользователях НЕ прогонялся автономно (destructive) — проверены только
  корректность команд и логика; перед боевым включением стоит контролируемый live-тест на одном конфиге.
- openvpn `disable` для активной долгой сессии применяется на реконнекте (не рвём текущие сессии).
