# 07 — Миграция сервера на новый VPS

## Задача

Старый VPS заблокирован/умер (или переезд плановый) — перенести сервер панели на новый хост:
сменить SSH-реквизиты и восстановить те же протоколы, по возможности сохранив валидность
выданных клиентских конфигов.

## Главный вывод (честность)

**Переносимость «с тем же материалом» сегодня невозможна ни для одного вендора**, потому что
`ServerProtocol.material_encrypted` хранит только ПУБЛИЧНЫЙ материал, достаточный для сборки
клиентских артефактов (build_artifact), но не приватную identity сервера. Приватные ключи
генерятся внутри контейнера при install (скриптами `setup_container` / `install_server.sh`)
и никогда не читаются обратно в панель:

| Вендор / протокол | Что хранится в material_encrypted | Чего НЕ хватает для переноса identity | Итог при миграции |
|---|---|---|---|
| awg (Amnezia, AWG2) | `server_public_key`, `psk` | приватный ключ сервера (генерится в контейнере) | новая пара ключей + psk → конфиги невалидны, **перевыдача** |
| awg_legacy (AWG 1.5) | `server_public_key`, `psk` | то же, что awg | **перевыдача** |
| xray (VLESS+Reality tcp) | `xray_public_key`, `short_id`, `bootstrap_uuid`, `site` | Reality **privateKey** (генерится `xray x25519` в контейнере) | новый pbk в vless:// → **перевыдача** (uuid клиентов можно было бы пере-добавить — см. remaining) |
| xray_xhttp (VLESS+Reality xhttp) | то же + `xhttp_path` | то же + путь XHTTP генерится заново | **перевыдача** |
| openvpn | `ca_cert`, `ta_key`, `transport` | приватный ключ CA + серверный серт (easyrsa PKI в контейнере) | новый CA не доверяет старым клиентским сертам и наоборот → **перевыдача** |
| outline | `outline_api_url`, `outline_cert_sha256` | всё состояние shadowbox (ключи) — на томе СТАРОГО хоста; apiUrl содержит старый IP | новые access-keys (`method:password`) → **перевыдача** |
| hysteria2 | `hysteria_obfs_password`, `hysteria_cert_sha256`, `site` | self-signed серт + obfs генерятся в контейнере; в hysteria2:// зашит pin серта | новый pinSHA256 → **перевыдача** (id+пароль клиентов хранятся у панели — пере-добавление возможно, см. remaining) |

Дополнительно: **endpoint во всех клиентских конфигах содержит IP сервера** — даже при полном
переносе identity смена IP всё равно требует перескачивания конфига пользователем. Поэтому
«прозрачно» (без действий пользователя) не переносится ничего; сохранение identity лишь
избавило бы от пересоздания клиентов на сервере (тот же uuid/pubkey, перескачивание вместо
перевыпуска).

## Что реализовано (MVP)

- `ServerService.migrate(owner_id, sid, data)` (`backend/src/vpnhub/services/servers.py`):
  - валидация нового IP (`is_valid_host`), опционально порт/юзер/способ авторизации/секрет
    (пустой секрет → остаётся текущий: тот же SSH-ключ может подходить к новому VPS);
  - одна транзакция: новые реквизиты, `status=unknown`, все installed provisioned-протоколы →
    `mark_installing`, `pending_revoke_json=None` (ledger-долг относился к контейнерам старого
    хоста), все активные `DeviceConfig` сервера → `status="revoked"`;
  - после транзакции — `schedule_install` по каждому вендору (существующий фоновый install-путь:
    прогресс через `state=installing` + SSE, ошибки per-protocol c `error_code`/remediation);
  - перевыдача конфигов — существующим flow: `configs._find_config` пропускает revoked →
    участник просто заново скачивает конфиг (уже с новым адресом и новым материалом), member-UI
    (`Devices.tsx`) уже показывает пометку «отозван».
- Эндпоинт владельца: `POST /api/v1/servers/{sid}/migrate` (`api/routers/owner.py`),
  ответ: `{ server, reinstall: {vendor: [proto]}, configsRevoked }`.
- UI (`ServerDetail.tsx` + `lib/queries.ts`): кнопка «Мигрировать на новый VPS» в карточке
  «Подключение SSH» → модалка (IP, юзер/порт, способ авторизации, секрет с подсказкой
  «пусто — использовать текущий») с предупреждением о перевыдаче; прогресс — существующие
  бейджи установки протоколов.
- Тесты (`backend/tests/integration/services/test_server_migration.py`, in-memory SQLite,
  schedule_install подменён): смена реквизитов + installing + revoke конфигов + обнуление
  ledger; сохранение секрета при пустом вводе; невалидный IP → BadRequest без побочек.
- Изменений моделей нет → Alembic-миграция не нужна (`revoked` уже допустимое значение
  `DeviceConfig.status`).

## Remaining (осознанно за рамками первого среза)

- **Перенос identity сервера**: начать персистить приватный материал при install
  (awg-приватник, Reality privateKey, PKI easyrsa, серт+ключ hysteria2) — тогда миграция
  сможет восстанавливать контейнер с тем же материалом (restore-файлы перед
  `setup_container`), и клиентам достаточно перескачать конфиг (тот же ключ/uuid, новый IP).
  Outline потребует бэкапа тома shadowbox (`/opt/outline`).
- **Пере-добавление клиентов с сохранением client_id**: для xray (uuid) и hysteria2
  (id+пароль хранятся у панели) можно после переустановки заново завести тех же клиентов —
  чтобы конфиг «оживал» перескачиванием без пересоздания DeviceConfig; для awg нужен
  клиентский pubkey (хранится), для openvpn — повторная подпись невозможна без CSR.
- **Сценарий «старый сервер жив»** (плановый переезд): вычистка контейнеров на старом хосте
  (снос по старым реквизитам перед переключением), опционально даунтайм-less порядок.
- **Автоперевыдача конфигов / нотификация участников**: сейчас участник узнаёт о перевыдаче
  по пометке «отозван» в устройствах; пуш/письмо и кнопка «перевыпустить всё» — отдельно.
- Аудит-событие миграции (`audit_types`) и отражение сводки миграции в Events.
- Протоколы в состоянии `error` на момент миграции не переустанавливаются автоматически —
  доустановка существующей кнопкой install/fix.
